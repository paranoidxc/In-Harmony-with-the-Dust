第 1 步：启动一个新对话，喂给 Claude 全局规则（一次性）
做法：在对话的第一条消息中，发送以下内容，只发一次，Claude 会记住整段对话：

```
接下来的所有对话，请严格遵循以下工程规范：

技术栈：
- 后端：Go + Gin + GORM + MySQL
- 前端：Vue 3 + Vuetify 3（仅限 CDN 引入，无构建工具，禁止 npm install）

流程铁律：
- 任何新功能需求，必须先输出一份 Spec 文档，包含：
  1. 用户故事
  2. 验收标准（Given-When-Then 格式）
  3. Out of Scope（明确这次不做什么，划清边界）
- 等我明确回复“确认”或“通过”后，你才能开始生成代码。
- 生成代码时，后端注意处理所有 error，禁止循环内查 DB（防 N+1）；前端使用 Vue 3 组合式 API，使用 CDN 全局变量。
- 代码完成后，请附带一段 MR 描述草稿，列出 API 变更、手动测试步骤、未改动的模块声明。

补充提交规范：
当我明确说“测试通过，请生成 Git message”时，请按以下格式输出（重点：不要操作git，只提供给我文本，我自己操作）：

标题 (Subject)：<type>(<scope>): <简短描述>，长度不超过 50 字符。type 取 feat(新功能)、fix(修复)、refactor(重构)、perf(性能)。

正文 (Body)：换行后，用列表列出本次改了啥（Why 和 What）。

脚注 (Footer)：如果有关联的 Issue 或需求单号，留空让我自己填。

```

--------------------

第 2 步：提出具体需求，要求只出 Spec
接下来，在同一个对话中发送你的需求：

```
现在，请针对以下需求输出 Spec（先不要写代码）：

需求：合同模块相关，计划开始时间和计划结束时间改为非必填项。计划结束时间为空时，不做到期提醒。

```

--------------------

第 3 步：人工审阅 Spec，确认或修改
阅读 Spec 内容，如果满意，直接回复：

```
确认，Spec 通过。

```
如果想修改，直接说“把 XX 改一下”，Claude 会更新 Spec，直到你确认。

--------------------

第 4 步：让 Claude 生成代码（利用上下文记忆）

```
请根据刚才确认的 Spec 生成完整的代码。包括：
- Go Gin 后端 handler 和 service
- Vue3 前端页面/组件（CDN 方式）
- 完成后附 MR 描述（API 变更、手动测试步骤、未改动模块声明）

```
Claude 会基于记忆的 Spec 生成代码，不会因为重复发送而 token 超长。

--------------------

第 5 步：检查 MR 描述和代码，如有问题继续调整

如果代码有偏差，可以直接说“XX 函数逻辑不对，请修改”，Claude 会在上下文中修正。

如果满意，你就拿代码去实现，整个流程结束。
